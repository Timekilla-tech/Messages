\section{Системийн архитектур}
Төслийн архитектурыг дараах гол хэсгүүдэд хувааж болно:
\begin{itemize}
    \item \textbf{Filtering Engine}: Ирж буй зурвасыг ``Intercept'' хийж, \texttt{Category} дүрмүүдийн дагуу \texttt{folder\_id} оноох логик.
    \item \textbf{UI Layouts}: RecyclerView-ийн Sticky Headers (Age Grouping) болон Foldable хоёр баганат харагдац.
    \item \textbf{Data Management}: Room v13 сангийн өргөтгөл болон SharedPreferences (JSON Mapping) ашигласан хавтаслах систем.
\end{itemize}

\textbf{Диаграммын бүрэлдэхүүн хэсгүүдийн тайлбар:}
\begin{itemize}
    \item \textbf{Data Layer (Өгөгдлийн давхарга)}: Room DB (v13) болон DAO (Data Access Objects) хэсгээс бүрдэх бөгөөд зурвас, ангилал, хавтасны мэдээллийг SQLite форматаар локал төхөөрөмж дээр хадгална.
    \item \textbf{Repository / Context Extensions}: Төслийн бизнес логикийн цөм хэсэг. Ирж буй зурвасыг Regex-ээр шалгаж ангилах (Filtering Engine) болон өгөгдлийн урсгалыг (Flow) удирдах үйлдлүүд энд ажиллана.
    \item \textbf{Presentation Layer (Танилцуулга давхарга)}: ViewModel, Activity болон Compose UI хэсгүүдийг агуулна. ViewModel нь өгөгдлийг UI-д харуулахад бэлэн ``State'' (төлөв) болгон хувиргаж, дэлгэц нугалах үед (Continuity) өгөгдлийг алдалгүй хадгалах үүрэгтэй.
\end{itemize}

\section{Өгөгдлийн загварчлал}
Системийн өгөгдлийн бүтцийг Room v13 схемд шилжүүлж, шүүлтүүрийн чадамжийг нэмэгдүүлсэн. Өгөгдлийн сангийн хувилбаруудын өөрчлөлтийг дараах байдлаар дүгнэж болно:

\begin{itemize}
    \item \textbf{Room v11}: Ангиллын системийн хамгийн анхны хувилбар бөгөөд зөвхөн \texttt{id}, \texttt{name}, \texttt{color} гэсэн үндсэн мэдээллүүдийг хадгалдаг байсан.
    \item \textbf{Room v12}: Ангилалд \texttt{keywords} болон \texttt{keyword\_is\_regex} гэсэн талбаруудыг нэмж, анхны автомат шүүлтүүрийн системийг нэвтрүүлсэн.
    \item \textbf{Room v13}: Шүүлтүүрийг илүү уян хатан болгохын тулд \texttt{plain\_keywords} болон \texttt{regex\_patterns} гэсэн хоёр тусдаа талбарыг нэмж оруулсан.
\end{itemize}

\begin{lstlisting}[language=Kotlin, caption=Category Entity-ийн шинэчлэгдсэн бүтэц (v13)]

\section{Ангиллын логик ба Regex боловсруулалт}
Системийн хамгийн чухал хэсэг болох зурвас ангилах логикийг \texttt{isMessageMatchingCategory} функцэд хэрэгжүүлсэн.

\begin{lstlisting}[language=Kotlin, caption=Зурвасыг Regex болон түлхүүр үгээр шалгах функц]
fun isMessageMatchingCategory(message: Message, category: Category): Boolean {
    val body = message.body.lowercase()
    val sender = message.senderPhoneNumber

    // 1. Энгийн түлхүүр үгээр шалгах
    if (category.plainKeywords.isNotEmpty()) {
        val keywords = category.plainKeywords.split(",").map { it.trim() }
        if (keywords.any { body.contains(it) || sender.contains(it) }) return true
    }

    // 2. Regex паттернээр шалгах
    if (category.regexPatterns.isNotEmpty()) {
        val patterns = category.regexPatterns.lineSequence()
            .map { it.trim() }.filter { it.isNotEmpty() }

        for (pattern in patterns) {
            try {
                val regex = Regex(pattern, RegexOption.IGNORE_CASE)
                if (regex.containsMatchIn(body) || regex.containsMatchIn(sender)) return true
            } catch (e: Exception) { continue }
        }
    }
    return false
}
\end{lstlisting}

\section{Foldable төхөөрөмжийн дэмжлэг}
Нугалах дэлгэцтэй (Foldable) төхөөрөмжүүд дээр хэрэглэгчийн интерфейсийг динамикаар өөрчлөхөд \texttt{WindowInfoTracker} санг ашигласан.

\begin{lstlisting}[language=Kotlin, caption=Foldable төлөвийг мэдэрч буй WindowInfoTracker код]
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    val tracker = WindowInfoTracker.getOrCreate(this)
    lifecycleScope.launch {
        tracker.windowLayoutInfo(this@MainActivity)
            .collect { layoutInfo ->
                onWindowLayoutInfoChanged(layoutInfo)
            }
    }
}

private fun onWindowLayoutInfoChanged(layoutInfo: WindowLayoutInfo) {
    val isLarge = layoutInfo.displayFeatures.any {
        it is FoldingFeature && it.state == FoldingFeature.State.FLAT
    }
    updateTwoPaneMode(isLarge)
}
\end{lstlisting}

\section{Хөгжүүлэлтийн явц ба Git бүтэц}
Төслийн хөгжүүлэлтийг системтэй, алдаа багатай гүйцэтгэхийн тулд Git-ийн ``Feature Branch'' стратегийг ашигласан.

\begin{itemize}
    \item \textbf{feature/* салбарууд}: Функц бүрийг тусгаарлан хөгжүүлсэн (\texttt{feature/regex}, \texttt{feature/fold}, \texttt{feature/category} г.м).
    \item \textbf{merged салбар}: Хөгжүүлж дууссан функцүүдийг нэгтгэсэн орчин.
    \item \textbf{test салбар}: Төхөөрөмж дээрх бодит туршилт болон нэгж тестүүдийг ажиллуулах салбар.
    \item \textbf{main салбар}: Баталгаажсан эцсийн хувилбар.
\end{itemize}

\section{Тест ба чанарын баталгаажуулалт}
Нийт 24 нэгж тест амжилттай хийгдэж, дараах функцүүдийг баталгаажуулсан:
\begin{itemize}
    \item \textbf{Category Matching (13 тест)}: Regex болон энгийн үгээр шүүх логик.
    \item \textbf{Keyword Parsing (5 тест)}: Түлхүүр үгсийг цэвэрлэх, Regex шалгах.
    \item \textbf{Foldable Helper (4 тест)}: Дэлгэцийн төлөвийг зөв тодорхойлох.
    \item \textbf{Age Buckets (2 тест)}: Мессежүүдийг хугацаагаар бүлэглэх.
\end{itemize}

Системийн ачаалал даах чадварыг 40,000 харилцан яриа, 1,000 энгийн үг, 1,000 Regex паттерн ашиглан шалгахад тогтвортой ажилласан. Туршилтын явцад тогтоосон хязгаарлалтууд:
\begin{itemize}
    \item Үүсгэж болох фолдерын дээд хязгаар: 4.
    \item Үүсгэж болох ангиллын тоо: 16.
    \item Ангилал бүрт оруулах үг, паттерны тоо: 100/100.
\end{itemize}

\texttt{MigrationTest.kt} ашиглан v10-аас v13 хүртэлх шилжилтийг 5,000+ мессеж дээр дата алдагдалгүй шалгаж дууссан.
