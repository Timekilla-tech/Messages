\chapter{Системийн зохиомж ба хөгжүүлэлт}

\section{Системийн архитектур}
Төслийн архитектурыг дараах гол хэсгүүдэд хувааж болно:
\begin{itemize}
    \item \textbf{Filtering Engine}: Ирж буй зурвасыг ``Intercept'' хийж, \texttt{Category} дүрмүүдийн дагуу \texttt{folder\_id} оноох логик.
    \item \textbf{UI Layouts}: RecyclerView-ийн Sticky Headers (Age Grouping) болон Foldable хоёр баганат харагдац.
    \item \textbf{Data Management}: Room v13 сангийн өргөтгөл болон SharedPreferences (JSON Mapping) ашигласан хавтаслах систем.
\end{itemize}

\textbf{Диаграммын бүрэлдэхүүн хэсгүүдийн тайлбар:}
\begin{itemize}
    \item \textbf{Data Layer (Өгөгдлийн давхарга)}: Room DB (v13) болон DAO (Data Access Objects) хэсгээс бүрдэх бөгөөд зурвас, ангилал, хавтасны мэдээллийг SQLite форматаар локал төхөөрөмж дээр хадгална.
    \item \textbf{Repository / Context Extensions}: Төслийн бизнес логикийн цөм хэсэг. Ирж буй зурвасыг Regex-ээр шалгаж ангилах (Filtering Engine) болон өгөгдлийн урсгалыг (Flow) удирдах үйлдлүүд энд ажиллана.
    \item \textbf{Presentation Layer (Танилцуулга давхарга)}: ViewModel, Activity болон Compose UI хэсгүүдийг агуулна. ViewModel нь өгөгдлийг UI-д харуулахад бэлэн ``State'' (төлөв) болгон хувиргаж, дэлгэц нугалах үед (Continuity) өгөгдлийг алдалгүй хадгалах үүрэгтэй.
\end{itemize}

\section{Өгөгдлийн загварчлал}
Системийн өгөгдлийн бүтцийг Room v13 схемд шилжүүлж, шүүлтүүрийн чадамжийг нэмэгдүүлсэн. Өгөгдлийн сангийн хувилбаруудын өөрчлөлтийг дараах байдлаар дүгнэж болно:

\begin{itemize}
    \item \textbf{Room v11}: Ангиллын системийн хамгийн анхны хувилбар бөгөөд зөвхөн \texttt{id}, \texttt{name}, \texttt{color} гэсэн үндсэн мэдээллүүдийг хадгалдаг байсан. Энэ үед мессежүүдийг автоматаар ангилах шүүлтүүрийн логик хараахан нэмэгдээгүй байв.
    \item \textbf{Room v12}: Ангилалд \texttt{keywords} болон \texttt{keyword\_is\_regex} гэсэн талбаруудыг нэмж, анхны автомат шүүлтүүрийн системийг нэвтрүүлсэн. Гэвч энэ нь хэрэглэгчийг зөвхөн нэг төрлийн (текст эсвэл Regex) шүүлтүүр ашиглахад хязгаарлаж байв.
    \item \textbf{Room v13}: Шүүлтүүрийг илүү уян хатан болгохын тулд \texttt{plain\_keywords} (энгийн үгс) болон \texttt{regex\_patterns} (төвөгтэй илэрхийллүүд) гэсэн хоёр тусдаа талбарыг нэмж оруулсан. Ингэснээр нэг ангилалд энгийн үг болон Regex-ийг зэрэг ашиглах боломж бүрдсэн.
\end{itemize}

\begin{lstlisting}[language=Kotlin, caption=Category Entity-ийн шинэчлэгдсэн бүтэц (v13)]
@Entity(tableName = "categories")
data class Category(
    @PrimaryKey(autoGenerate = true) var id: Long = 0,
    var name: String,
    var color: Int,
    var plainKeywords: String = "", // Шинээр нэмэгдсэн
    var regexPatterns: String = ""  // Шинээр нэмэгдсэн
)
\end{lstlisting}

Өгөгдлийн сангийн шилжилт (Migration)-ийн явцад хуучин \texttt{keywords} талбарт байсан өгөгдлийг алдалгүйгээр шинэ талбарууд руу хөрвүүлэх логикийг \texttt{Migration(12, 13)} хэсэгт хэрэгжүүлсэн.

\section{Хөгжүүлэлтийн явц ба Git бүтэц}
Төслийн хөгжүүлэлтийг системтэй, алдаа багатай гүйцэтгэхийн тулд Git-ийн ``Feature Branch'' стратегийг ашигласан. Энэ нь шинэ функц бүрийг тусдаа салбар (branch) дээр хөгжүүлж, туршсаны дараа үндсэн кодтой нэгтгэх боломжийг олгосон.

\begin{itemize}
    \item \textbf{feature/* салбарууд}: Функц бүрийг тусгаарлан хөгжүүлсэн (\texttt{feature/regex}, \texttt{feature/fold}, \texttt{feature/category} г.м).
    \item \textbf{merged салбар}: Хөгжүүлж дууссан функцүүдийг нэгтгэж, хоорондын уялдааг шалгах орчин.
    \item \textbf{test салбар}: Төхөөрөмж дээрх бодит туршилт болон нэгж тестүүдийг ажиллуулах салбар.
    \item \textbf{main салбар}: Тогтвортой, алдаагүй ажиллагаа нь баталгаажсан эцсийн хувилбар.
\end{itemize}

Энэхүү бүтэц нь олон функц зэрэг хөгжүүлэгдэж буй үед кодын зөрчил (conflict) үүсэхээс сэргийлж, хувилбарын удирдлагыг илүү тодорхой болгосон.

\section{Тест ба чанарын баталгаажуулалт}
Системийн тогтвортой ажиллагаа болон логик алдааг шалгах зорилгоор нэгж тестүүдийг (Unit Tests) тогтмол ажиллуулж байна. Одоогоор нийт 24 нэгж тест амжилттай хийгдэж, дараах үндсэн функцүүдийг баталгаажуулсан:

\begin{itemize}
    \item \textbf{Category Matching (13 тест)}: Regex болон энгийн үгээр шүүх логик, legacy форматын дэмжлэг, олон мөртэй мессежийг шүүх чадвар.
    \item \textbf{Keyword Parsing (5 тест)}: Оруулсан түлхүүр үгсийг цэвэрлэх (normalize), давхардлыг арилгах, Regex-ийн хүчинтэй байдлыг шалгах.
    \item \textbf{Foldable Helper (4 тест)}: Дэлгэцийн хэмжээг зөв тодорхойлж, Foldable болон Tablet төхөөрөмж дээр split-view-г зөв идэвхжүүлэх.
    \item \textbf{Age Buckets (2 тест)}: Мессежүүдийг хугацаагаар нь (Өнөөдөр, Өчигдөр, Өнгөрсөн сар г.м) зөв бүлэглэх.
\end{itemize}

Туршилтаар бүх тестүүд амжилттай (Passed) гарсан бөгөөд системд ноцтой логик алдаа байхгүй болохыг баталлаа.

Системийн гүйцэтгэл болон тогтвортой байдлыг шалгах зорилгоор 40,000 харилцан яриа (conversations), 1,000 энгийн түлхүүр үг (plain words), болон 1,000 Regex паттерн ашиглан ачааллын туршилт (Stress Test) хийхэд систем ямар нэгэн crash (гацалт) өгөөгүй тогтвортой ажилласан.

Туршилтын явцад 5-аас дээш фолдер үүсгэх үед системийн ажиллагаанд алдаа зааж байсан тул системийн тогтвортой байдлыг хангах үүднээс дараах техникийн хязгаарлалтуудыг тогтоож өгсөн:
\begin{itemize}
    \item Үүсгэж болох фолдерын дээд хязгаарыг 4 болгож хязгаарласан.
    \item Үүсгэж болох ангиллын тоог нийт ялгаатай өнгийн тоогоор буюу 16-аар хязгаарласан.
    \item Ангилал бүрт оруулах энгийн үг болон Regex паттерны тоог тус бүр 100 болгон хязгаарлаж, санах ойн ачааллыг оновчилсон.
\end{itemize}

Мөн \texttt{MigrationTest.kt} ашиглан v10-аас v13 хүртэлх шилжилтийг шалгасан бөгөөд 5,000+ мессежтэй өгөгдлийн санг шилжүүлэхэд data loss үүсээгүй байна.
